# Методология «Бренд-платформа» 3.2

> Источник истины для продуктового канона BP-платформы.
> Дата: 18.04.2026. Автор канона: А.В. Чирков.
> Платформенная адаптация (перенос с рабочей таблицы на wizard): BP team.

---

## Зачем этот документ

Стандарт 3.1 (docx, 16.04.2026) описывал процесс в формате рабочей таблицы Google Sheets из 6 листов. Версия 3.2 фиксирует перенос того же процесса на программную платформу — **без изменения методологии**. Именно поэтому мажорный номер стандарта не меняется (3.1 → 3.2 — patch, не новая методология).

**Инвариант:** методология — moat. Технология — инструмент. Любая платформенная адаптация должна сохранять смысл 3.1, но использовать то, что даёт платформа (LLM-черновики, блокировки, version history, audit trail).

---

## Что изменилось с 3.1 → 3.2

### Процесс остался тем же

- 4 шага (голос клиента → сессия → архетип и позиционирование → тесты месседжа)
- 8–12 рабочих дней
- 1 сессия с собственником на 3 часа (не 4 встречи)
- Топ-3 похвалы / топ-3 нарекания / ключевой инсайт — по-прежнему требуемый результат Стадии 1
- 5 канонических вопросов интервью — те же, в текущей формулировке

### Что заменяет платформа

| Было (3.1)                                | Стало (3.2 на платформе)                                                                                 |
| ----------------------------------------- | -------------------------------------------------------------------------------------------------------- |
| Рабочая таблица из 6 листов               | Wizard из 4 стадий (Стадия 1 = Лист 1; Стадия 2 = Лист 4; Стадия 3 = Листы 5–6; Стадия 4 = тесты Лист 6) |
| Маркетолог сам пишет сводку по интервью   | Claude извлекает паттерны, маркетолог **редактирует и утверждает**                                       |
| Маркетолог вручную ищет повторения        | LLM считает `occurrences` по смысловому совпадению (≥2 вхождений)                                        |
| Утверждение с собственником в переписке   | `Approval` с immutable snapshot + audit-trail (`generated_by` / `modified_by` / `approved_by`)           |
| Версия стандарта на титульной странице    | Автоинкремент patch при каждом билде (`PLATFORM.version` в `config/platform.ts`)                         |

### Что платформа принципиально не меняет

- **Claude — черновик, человек — утверждение.** Ни одна стадия не «закрывается» автоматически. Кнопка «утвердить и перейти к следующей стадии» — явный consent маркетолога (и, где применимо, собственника).
- **Порядок стадий зафиксирован.** Стадия N+1 не открывается пока Стадия N не `approved`. Это архитектурный инвариант, не UX-рекомендация.
- **Голос клиента первичен.** Стадия 2 (сессия с собственником) опирается на утверждённый результат Стадии 1. Собственник получает готовый список цитат клиента и отвечает через призму этих цитат.

## Стадия 1 — «Портрет клиента» (бывший Лист 1)

### Входные данные

- 3–5 транскриптов интервью (целевая цифра — 5; минимум 3 для разблокировки «Извлечь паттерны»).
- Индустрия проекта (для подбора отраслевых gotchas).

### 5 канонических вопросов (не меняются)

1. «Если бы мы закрылись завтра, чего конкретно вам бы не хватило?» — вскрывает реальную ценность.
2. «Почему вы выбрали нас, а не конкурента?» — реальное позиционирование в голове клиента.
3. «Как бы вы описали нас другу в одном предложении?» — черновик бренд-месседжа.
4. «Что вас раздражает в нашей нише в целом?» — боли рынка для отстройки.
5. «Был ли момент, когда вы подумали „вот это круто"?» — пиковые моменты.

Маркетолог записывает **дословно**. Платформенная подсказка напоминает: не пересказывать своими словами, вставлять прямую речь.

### Результат Стадии 1 (обновлённая схема)

Каноническая выжимка 3.1 — «топ-3 похвалы, топ-3 нарекания, ключевой инсайт» — **обязательны**. На платформе добавляются структурированные поля для автоматического переноса в Стадию 2 и далее:

| Поле                      | Что это                                                                                           | Канон 3.1 |
| ------------------------- | ------------------------------------------------------------------------------------------------- | --------- |
| `key_insight`             | Одна фраза: главный вывод из всех интервью                                                        | ✅         |
| `top_praise`              | Топ-3 формулировки, за что хвалят (отсортированы по частоте)                                      | ✅         |
| `top_criticism`           | Топ-3 формулировки, за что ругают / что раздражает в нише                                         | ✅         |
| `pains`                   | Все зафиксированные боли с `occurrences` и цитатами (для Стадии 3, отстройка от конкурентов)       | расширение |
| `gains`                   | Атрибуты опыта, которые радуют (для бренд-ценностей на Стадии 2)                                   | расширение |
| `jobs_to_be_done`         | JTBD в каноническом формате: `when [ситуация] → want [действие] → so that [исход]`                 | расширение |
| `repeated_phrases`        | Дословные выражения клиента — кандидаты для бренд-месседжа                                         | расширение |
| `emotional_markers`       | Эмоциональные слова (страх, гордость, облегчение), **без цитатных пересечений** с `repeated_phrases` | расширение |
| `segment_signals.demography`           | Возраст, пол, семейный статус, география                                  | расширение |
| `segment_signals.acquisition_channels` | Каналы привлечения: рекомендация, отзывы, реклама, органика               | расширение |
| `segment_signals.price_markers`        | Ценовые маркеры: средний чек, отношение к скидкам, порог отказа           | расширение |
| `segment_signals.behavior_patterns`    | Поведение при выборе: сравнивает цены, читает отзывы, приходит по знакомству | расширение |

**Правило дедупликации.** Одна цитата может появиться в **максимум двух** блоках (например, в `repeated_phrases` как фраза и в `quotes_for_owner_session` как источник). Цитата не может одновременно быть в `repeated_phrases` и `emotional_markers` — выбирается доминирующая функция (язык для копирайтинга vs эмоциональный маркер для анализа).

**Правило сортировки.** Все списки внутри блоков (`pains`, `gains`, `jobs_to_be_done`, `repeated_phrases`, `emotional_markers`) отсортированы по `occurrences` **по убыванию**. Топ-3 в каждом блоке визуально выделены.

**Правило JTBD-формулировки.** Формат: `когда [ситуация] → хочу [действие] → чтобы [исход]`. Формулировка типа «получить предсказуемость» — это gain, не job. Пример правильного JTBD: «Когда у ребёнка болит зуб в выходные → хочу найти клинику в тот же день → чтобы ребёнок не провёл 48 часов в страхе и боли».

### DoD Стадии 1

Маркетолог может нажать «Утвердить и перейти к Стадии 2» только когда:

1. Загружено ≥3 интервью (на практике — 5).
2. Сгенерирован черновик паттернов (`ai.json` с не-null полями).
3. Маркетолог либо согласен с черновиком, либо прислал структурированные правки и перегенерировал.
4. `key_insight` заполнен и не пустой.

После подписи — `row.status = 'approved'`, создаётся `Approval` снимок (immutable, 7 лет object-lock в S3), Стадия 2 разблокируется.

## Стадия 2 — «Сессия с собственником» (бывшие части Листа 4)

(детально — не в рамках этого документа, но для полноты контекста)

- Блок 1: легенда (60 мин)
- Блок 2: ценности + тест на увольнение + 4 компонента на каждую ценность (60 мин)
- Блок 3: миссия + видение будущего (60 мин)

**Вход в Стадию 2:** утверждённый результат Стадии 1 (`key_insight`, `top_praise`, `top_criticism` + полный набор цитат).

**Принципиальное изменение от 3.1:** собственник не начинает «с чистого листа». Перед сессией Claude готовит для него документ «Что сказали ваши клиенты» — цитаты, сгруппированные по топ-3 / по ценностным пересечениям / по страхам. Собственник на сессии отвечает через призму этого документа. Это сокращает сессию с 4-х часов до 3-х и снимает риск самообмана собственника.

---

## Стадия 3 — «Архетип и позиционирование» (бывшие Листы 5–6)

- Архетип: один из 6 (Заботливый / Мудрец / Бунтарь / Искатель / Правитель / Простодушный).
- Таблица «Мы / Мы не» — 6 строк.
- Позиционирование по формуле: «Для [аудитория], которая [потребность], [название] — это [категория], которая [ключевая выгода], потому что [доказательство]».
- Бренд-месседж: 4–7 слов, сгенерированы по 3 техникам (противопоставление / метафора / результат).

**Выбор архетипа — judgment-heavy**: платформа показывает рекомендацию LLM, но финальный выбор делает маркетолог вместе с собственником. Платформа не создаёт иллюзию «AI решил архетип».

---

## Стадия 4 — «4 теста месседжа» (бывший Лист 6, нижняя часть)

1. Тест клиента — спросить 5 клиентов из Стадии 1.
2. Тест сотрудника — какую фразу говорят друзьям.
3. Тест памяти — показать 5 вариантов, через час спросить что запомнили.
4. Тест гордости — представить фразу на визитке: хочется раздавать или неловко?

**Отличие от 3.1:** платформа автоматически готовит черновик письма для клиентского теста (с цитатами клиента) и протокол для теста сотрудника. Маркетолог редактирует и рассылает.

## Ошибки и анти-паттерны (из 3.1 — остаются в силе)

| Ошибка                                      | Решение на платформе                                                                             |
| ------------------------------------------- | ------------------------------------------------------------------------------------------------ |
| Начать с собственника, а не с клиента        | Стадия 2 заблокирована до approve Стадии 1                                                        |
| Абстрактные ценности без поведения           | На Стадии 2 — обязательные 4 компонента каждой ценности + тест на увольнение                       |
| Позиционирование без карты конкурентов       | Отдельный раздел Стадии 3, блокирует формулировку позиционирования                                 |
| Разрыв обещания и реальности                 | Стадия 4 (4 теста) — обязательна, DoD = все 4 теста пройдены                                       |
| Платформа создана без команды                | В `Approval` отдельный slot для мнений команды (`team_feedback`)                                 |
| Откладывание интервью с клиентами            | Блокировка Стадии 1 на 3 интервью минимум; напоминание «начните с 5, не с 15»                     |

## Связка с другими блоками программы Чиркова

- **ID-Driver** (следующий блок, вне BP): отвечает на «КАК транслировать». Принимает на вход утверждённую бренд-платформу.
- **Блок 9 «Тендер»** и **Блок 1 «Аудиты»**: поставляют начальные данные по конкурентам. Платформа умеет импортировать их в Стадию 3 (фича планируется в 3.3).
