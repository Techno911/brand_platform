# ADR-0002. Не делаем low-code wizard builder (отклонение рекомендации)

- **Статус:** accepted, 2026-04-18
- **Автор:** Чирков
- **Касается:** архитектура wizard'а `backend/src/wizard/*` + `frontend/src/wizard/*`

## Контекст

Иван Смирнов в фидбэке 17.04.2026 предложил: сделать конфигурируемые
wizard-сценарии, чтобы chip_admin мог менять порядок стадий, добавлять
кастомные шаги, настраивать промпт-цепочки без релиза. Аргументация —
«гибкость под разные индустрии», «быстрее экспериментировать с методологией».

Это явная анти-рекомендация, и мы её отклоняем. Фиксируем аргументы
для будущих споров.

## Решение

**Wizard BP линейный и зашит в код.** 4 стадии, фиксированный порядок,
DoD-gated переходы. Промпт-шаблоны лежат в `backend/src/prompts/*.md` под
git, меняются через PR + golden-set regression gate.

Никакого low-code builder'а: ни JSON-DSL «шаги», ни drag-n-drop редактора,
ни «админ создаёт нового ассистента через форму».

## Почему

### 1. Методологический риск

Методология БП 3.1 — _законченный_ продукт Чиркова. Последовательность
(портрет клиента → сессия с собственником → архетип+позиционирование →
4 теста) не произвольная — это 18 лет консалтинга, собранные в
воспроизводимый pipeline. Low-code builder открывает дверь маркетологу
/ ассистенту _менять порядок_: «а давайте пропустим сессию с собственником,
он ведь уже в брифе всё написал». Это разрушает methodology-compliance.

Отдельный ADR-0001 вводит `methodology-compliance-check` как LLM-judge
по канону 3.1. Low-code builder сделает этот judge бесполезным: если
каждый проект может иметь свою структуру стадий, то judge не знает,
что сравнивать.

### 2. Moat — не в гибкости

Настоящий moat BP = **невозможность клиента воспроизвести это самому.**
Конфигурируемость через UI — ровно противоположное: если chip_admin
мышкой собирает wizard, то завтра это может сделать клиент Чиркова,
ушедший к конкуренту. Мы продаём _структуру_, а не _инструмент для
создания структур_.

Ср. с курсом «Ой-Май-AI», Роман Кумар Виас: «Вайпкодинг CRM vs SaaS —
не делайте CRM сами». Мы продаём методологию Чиркова, не платформу для
построения методологий.

### 3. Комбинаторный взрыв QA

Low-code означает: маркетолог X создал wizard с шагами A→B→C, маркетолог Y —
A→C→B→D. Golden set и regression-тесты должны покрыть _все_ комбинации,
иначе `/methodology-compliance-check` бесполезен.

Текущий подход: 4 стадии × 11 промптов × 3 вендора = 132 golden-set
комбинаций. Это уже много. Добавить «свободно конфигурируемые шаги» —
это экспоненциальный рост QA-нагрузки. Мы не SaaS для миллионов — у нас
10-30 клиентов. QA-штат будет 1 (Чирков) + Claude, не целая команда.

### 4. UX маркетолога = defaults, не настройка

Из конференции «Ой-Май-AI» (Хегай): маркетолог-гуманитарий не хочет
JSON / DSL / конфиги. Он хочет «нажать Next». Low-code builder — это
инструмент для chip_admin (то есть Чиркова), но Чирков _сам_ не хочет
каждую неделю настраивать wizard — это лишняя работа, он хочет
сосредоточиться на методологическом аудите пилотов, а не на lego.

Кастомизация на клиента происходит _не через wizard, а через industry_context_
(`backend/src/knowledge/industry_context/{industry}.md`): маркетолог
выбирает отрасль при создании проекта, Claude автоматически подгружает
контекст. Это единственная «гибкость», которая нужна.

### 5. Операционный риск

Если chip_admin мышкой собирает wizard — в любой момент он может
сломать его мышкой (случайно удалить шаг, переставить порядок).
Восстановление из audit log — возможно, но сложно, и клиент тем временем
сидит со сломанным wizard.

Текущий подход: изменение wizard = PR в git = ревью + CI + deploy.
Медленнее, но атомарно и обратимо.

## Что мы делаем ВМЕСТО low-code

- **Промпт-шаблоны в git под PR.** Любая кастомизация методологии — это
  изменение шаблона в `prompts/` с golden-set regression gate.
- **YAML-frontmatter у каждого шаблона** (`name`, `trigger`, `expected_output_schema`) —
  для метаданных без low-code runtime.
- **Per-industry контекст** (`industry_context/`, `industry_gotchas/`) —
  единственная точка кастомизации под клиента, без изменения pipeline.
- **Config per project** в БД (tariff, budget, model override через
  `forceVendor` на golden-set тестах) — это _параметры_, не _структура_.

## Последствия

### Плюсы (того, что не делаем low-code)
- Методология защищена от разрушения маркетологом/менеджером.
- Moat сохраняется.
- QA-матрица остаётся подъёмной.
- UX для маркетолога остаётся «нажать Next», как и требуется.

### Минусы (принятые)
- **Каждая новая индустрия = PR в prompts/ + industry_context/** — не
  «админ мышкой добавил». Это медленнее.
- **Эксперименты с порядком стадий невозможны через UI.** Хочешь проверить
  альтернативу — ответвляй golden-set, руки в код.
- **Chip_admin не может «на лету подкрутить» wizard для одного клиента.**
  Кастомизация — через industry_context, не через структуру wizard.

Эти минусы — не баги, это фича. Они защищают методологию от случайного
разрушения и от копирования.

## Связанные решения

- ADR-0001: multi-vendor LLM (мы _сменили вендора_ — это ок, потому что
  это инфраструктура, не методология).
- `CLAUDE.md §Принципы`: «Linear wizard, не автономный агент» — архитектурный
  инвариант, отражает это ADR.
