# Shadow AI Policy — Brand Platform

> **Назначение.** INSIGHTS §5 delta-7: корпоративная политика использования AI-инструментов
> персоналом Чиркова и внешними подрядчиками. Применимо к `chip_admin`
> (Чиркова) и всем маркетологам-подрядчикам (role `marketer`). Нарушение —
> дисциплинарное взыскание, для подрядчиков — разрыв контракта.
> **Версия 1.0.** Утверждено: 2026-04-17. Пересмотр раз в год или по мере изменения ландшафта.

---

## 1. Почему это важно

Бренд-платформа содержит:

- Персональные данные собственников-клиентов (голос в транскриптах, личные истории, семейная легенда)
- Коммерческую тайну клиентов (позиционирование, отстройка от конкурентов, разбор ЦА)
- Запатентованную методологию Чиркова 3.1 (внутренние критерии, эталоны, шаблоны)
- Reseller-договор с Anthropic (ключ, квоты, ценообразование)

**«Shadow AI»** — использование внешних LLM/chat-сервисов, не проходящих через нашу
reseller-инфраструктуру (ChatGPT.com, Gemini.google.com, Claude.ai, Perplexity,
локальная LLaMA с публичным API, Notion AI и т.п.).

Скармливание им клиентских данных = утечка. Даже если «я не сохранял историю» —
большинство сервисов хранит промпты для fine-tune, модератор может прочитать,
провайдер может быть взломан. Для нас это:

- Юридический риск (клиент подпишется NDA, мы его сливаем неустановленной стороне).
- Репутационный риск (клиент узнаёт → уходит → пишет в Tg-канал).
- Методологический риск (наши эталоны попадают в обучающий корпус чужой модели).

## 2. Что МОЖНО

### 2.1 Работа с данными клиентов

- Только через **Brand Platform** (`https://chirkov-bp.ru`). Внутри BP всё идёт
  через наш reseller-API к Anthropic с нашим промпт-кэшем, our-logs и retention 14 дней.
- При необходимости аналитики/summary **вне** платформы — **скопировать артефакт
  из BP в локальный `.md` и работать с ним без привлечения внешних LLM**.
- Для транскрипции звонков с собственником — **только встроенный в BP рекордер**
  (или Fireflies через одобренный нами workspace с подписанным DPA).

### 2.2 Работа с методологией и внутренней документацией

- Можно использовать **личный Claude Pro/Max** для вопросов общего характера:
  «как писать бренд-книгу», «какие бывают архетипы», «объясни разницу между
  позиционированием и миссией».
- **Нельзя** вставлять фрагменты методологии 3.1, эталоны, `industry_context/*`,
  `industry_gotchas/*` или golden set — это наш моат, не уходит вовне.

### 2.3 Работа с кодом BP (только для DEV-стороны Чиркова)

- Claude Code в IDE — **разрешён** при условии, что:
  1. Проектный `CLAUDE.md` загружен (архитектурные инварианты).
  2. Нет прямой отправки прод-секретов (`.env`, `prod.env`, certs) — в репе есть
     pre-commit hook на gitleaks.
  3. Не вставляется реальный `ANTHROPIC_API_KEY` — только заглушка `sk-ant-…`.
- **Запрещён** Cursor/Windsurf/VS Code Copilot с корпоративным аккаунтом если
  его опция «использовать код для обучения модели» не отключена и не
  подтверждена DPA с провайдером.

### 2.4 Обсуждение проекта в текстовом виде

- Внутренний Telegram-чат с ботом BP (`@chip_bp_bot`) — разрешён для статусов
  и пинг-уведомлений. Не сыпать туда транскрипты и черновики месседжа.
- Slack/Discord/личные мессенджеры — **запрещены** для обсуждения клиентских
  артефактов. Выносить обсуждение в секцию «коммуникации проекта» внутри BP
  (на этапе wizard'а) или в защищённую комнату Chirkov-WS.

## 3. Что НЕЛЬЗЯ (Hard Red Lines)

1. ❌ Загружать транскрипт сессии с собственником в любой внешний сервис.
2. ❌ Копировать целый `brand_message`, `positioning`, `legend`, `mission`,
   `values`, `vision` клиента в ChatGPT/Gemini/любой другой чат-бот.
3. ❌ Скидывать на внешние LLM готовый DOCX/XLSX-экспорт из BP.
4. ❌ Использовать публичные Google Docs/Notion-страницы с артефактами клиента
   без ACL-ограничения на команду Чиркова.
5. ❌ Сохранять артефакты в личные облака (Dropbox, Google Drive, iCloud),
   не связанные с корпоративным SSO Чиркова.
6. ❌ Обсуждать содержимое работы с клиентом в LinkedIn/Twitter без письменного
   согласования с Чирковым (даже намёком — «работаю с клиникой, интересная задача»).
7. ❌ Подключать MCP-серверы Notion/Slack/Google Drive к Claude Code на машине,
   где открыт репозиторий BP.
8. ❌ Публиковать в открытых репозиториях любые фрагменты `prompts/**` и
   `knowledge/**` (pre-commit проверяет + gitleaks).
9. ❌ Устанавливать неодобренные плагины Claude Code, Cursor, VS Code, которые
   читают файлы репозитория и отправляют их провайдеру.
10. ❌ Запускать `claude` CLI в неизолированном bash-окружении на проде (нет
    прода на чужом ноутбуке — только staging/dev).

## 4. Приёмка и контроль

- `chip_admin` ежемесячно проводит аудит:
  - Grep по логам BP на отклонения от reseller-ключа.
  - Проверка `security_events` таблицы (prompt injection, PII leak).
  - Проверка через Chrome Enterprise (для управляемых устройств) — наличие
    установленных запрещённых расширений.
- При обнаружении нарушения:
  1. Блокировка учётной записи в BP до разбора.
  2. Ретроспективный анализ: что было слито, какому провайдеру.
  3. Уведомление клиента (если затронуты его данные) в течение 72 часов
    (требование 152-ФЗ «О персональных данных»).

## 5. Технические подпорки

- CSP в ingress nginx (`img-src 'self'`) ограничивает загрузку скрытых пикселей.
- На уровне backend: `BriefSanitizerService` блокирует ввод, похожий на
  «скопированный из ChatGPT ответ» (эвристика: markdown, нумерованные списки
  структуры «1., 2., 3.», фразы «As an AI…»). Маркетолог получает warning.
- Все входящие и исходящие запросы `AnthropicClient` проходят через один
  `ai-gateway` модуль NestJS — там единый логгинг и budget kill-switch.
- Frontend Zustand хранит JWT access **только в памяти**. Ни один плагин
  браузера не сможет его прочитать через localStorage.

## 6. Исключения

Разовые исключения (например, вендор просит прислать скриншот диалога с Claude
Code для багрепорта) согласовываются лично с Чирковым через Telegram-канал
`@chip_bp_admin` с указанием:

- Что именно отправляется (полный текст / скриншот / файл)
- Кому (компания, email, роль)
- Срок удаления у получателя
- Подпись получателя о неразглашении

Журнал исключений — `backend/audit_log` (тип `shadow_ai_exception`).

## 7. Обучение

При онбординге:

1. Новый подрядчик/сотрудник читает этот документ **в первый день**.
2. Подписывает электронный акт ознакомления в BP → запись в `audit_log`.
3. Проходит короткий тест (10 минут, 7 кейсов «можно/нельзя»).
4. Дальше — раз в год, напоминание + мини-квиз.
