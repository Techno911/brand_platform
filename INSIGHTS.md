## 0. TL;DR — топ-10 действий на MVP (до пилота Оли)

1. **`BriefSanitizerService`** (input-sanitizer) — до первого LLM-вызова (vendor-agnostic; prompt injection — Top-1 уязвимость LLM). [→ §5]
2. **Kill-switch pre-flight budget check** на каждый LLM-вызов. Считает токены × price-per-token выбранного вендора. [→ §1]
3. **Prompt caching** для `industry_context/*.md` + knowledge base + шаблонов промптов. Per-vendor (Anthropic explicit `cache_control`, OpenAI auto-prefix, DeepSeek отдельное поле). [→ §1]
4. **`prompt_run` таблица** с `error_code` / `provider_latency_ms` / `retry_count` / `routingDecision` / `vendor` + Grafana + Telegram-бот `chip_admin`. [→ §1]
5. **Одна LLM-сессия на стадию** (не один гигантский промпт на весь wizard). [→ §1]
6. **Plan-mode «15 уточняющих вопросов по одному»** перед первой генерацией каждой стадии. [→ §3]
7. **Output validator 3-уровневый** (regex + LLM-judge + human) + `methodology-compliance-check`. Judge — на `tertiary` (дёшево). [→ §4]
8. **Golden set на 3-5 эталонов × 3 вендоров** (per-vendor регрессия; CI блокирует при >15%). [→ §4]
9. **Audit log с `responsible_user_id`** + `generated_by = ${vendor}:${model}` + `modified_by` / `approved_by` + immutable snapshot на `owner_approved`. [→ §5]
10. **Multi-vendor LLM-слой** (ADR-0001) — `VendorRouterService` + `GlobalLLMQueueService` + `ProjectBusyService`. Primary=Anthropic, secondary=OpenAI, tertiary=OpenAI-compat. Fallback chain на auth/rate_limit/transient/content_filter. [→ §1, ADR-0001]

## 1. Архитектура AI-слоя BP

### MVP

- **Kill-switch на стороне клиента (pre-flight budget check).** Перед каждым Opus-вызовом считаем запланированные input-токены + лимит output, сравниваем с бюджетом проекта. Если превышает — блокируем до отправки, возвращаем `BUDGET_EXCEEDED`. Живёт в `AIService` / NestJS middleware для 6 slash-команд. *Источник: курс, файл 10 — «блокировка на стороне клиента до отправки».*
- **Prompt caching** для системных промптов и knowledge base. `industry_context/*.md` + `industry_gotchas/{industry}.md` + промпт-шаблоны идут как `cache_control: ephemeral` блок. Критично для reseller-маржи (~10× экономии на повторах). *Источник: курс, файл 10 + навык claude-api.*
- **Структурированный JSON-лог каждого AI-вызова.** Поля: `input_tokens` / `output_tokens` / `stop_reason` / `cache_read_input_tokens` / `routing_decision` / `permission_denied` / `crash_reason` / `error_code` / `provider_latency_ms` / `retry_count`. Экспорт в Grafana + Telegram-нотификации для `chip_admin`. *Источник: курс, файлы 01 + 10; Твердохлебов «свистки, триггеры».*
- **Idempotency state на уровне задачи.** Каждая генерация (`ValueDraft`, `MissionVariant`, etc.) имеет статус `planned`/`executing`/`completed`/`failed`. Рестарт воркера с `executing` не пересчитывает — восстанавливает из last response. *Источник: курс, файл 10.*
- **Retry только для transient** (429, 5xx, timeout), 3 попытки с экспоненциальной задержкой. 4xx (кроме 429) — fail-fast. Реализация: NestJS interceptor в `AnthropicClient`. *Источник: курс, файл 01 — Alpaca pattern.*
- **Graceful degradation на падение Anthropic.** Wizard не блокируется — маркетолог видит баннер «ассистент недоступен, заполните вручную», работа сохраняется. *Источник: курс, файл 01.*
- **Один агент = один фокус задачи.** Отдельные Claude-сессии для каждой из 4 стадий (voice of customer / owner session / archetype+positioning / tests). *Источник: Горшков.*
- **Контекст = главное вложение.** `industry_context/{отрасль}.md` содержит десятки KB реальных данных (отзывы, конкуренты, эталоны, типовые боли). При Стадии 1 маркетолог выбирает отрасль → Claude подгружает контекст. *Источник: Замесин — «десятки килобайт про продукт и сегменты».*
- **Ролевые LLM-агенты per стадия.** `wizard-stage-orchestrator` регистрирует под каждый шаг отдельного агента с ролью (дешифровщик брифа / генератор черновиков / QA output validator / «CTO-ревьюер» от Чиркова). `chip_manager` видит канбан-доску задач стадии. *Источник: Твердохлебов — «пятеро агентов и внешний CTO».*

### С оговорками

- **`max_agent_roundtrips_per_stage = 5`.** Arbitration layer нужен, только если пойдём в multi-agent (критик + автор); иначе LLM могут пинг-понговать до исчерпания токенов. *Источник: Твердохлебов + Щербатюк «восстание машин».*
- **Sub-agents Claude Code.** В backend BP НЕ используем (мы — server-to-server Anthropic SDK, не CLI). Для dev-стороны (наша команда) полезны: `prompt-tester`, `golden-set-runner`. *Источник: курс, файл 11.*
- **Auto-compact контекста.** В BP нет длинных диалогов — каждая команда это один запрос. Включать только если появится «chat-уточнений» поверх wizard'а.

## 3. Slash-команды wizard'а (prompt engineering)

### MVP

- **Skill = обёртка методологии, не универсальный промпт.** Каждая из 6 команд (`/values-draft`, `/legend-draft`, `/mission-variants`, `/positioning-draft`, `/message-variants`, `/review-classify`) содержит полный пайплайн методологии 3.1, а не «сгенерь 5 вариантов». *Источник: Замесин — «скилл — это обёртка методологии».*
- **YAML-frontmatter метаданные** для каждого промпт-файла: `name`, `description`, `trigger_stage`, `expected_output_schema`, `max_tokens`, `cache_control`. Позволяет DEV-утилите `prompt-tester` загружать промпты декларативно. *Источник: курс, файл analysis (5).*
- **Plan-mode «15 уточняющих вопросов по одному».** Первая клетка каждой стадии — не генерация, а сбор уточнений от маркетолога Claude-агентом. Снижает халтуру на выходе. *Источник: Твердохлебов.*
- **`/critique-message` — 3 критика, 3 итерации.** На Стадии 3 (генерация бренд-месседжа) Claude прогоняет черновик через 3 роли: senior architect (логика месседжа), PM (соответствие ЦА), domain stakeholder (здравый смысл собственника). Каждая находит 3 ошибки → исправление → повтор 3 раза. *Источник: Елкин.*
- **`/challenge-owner-response` — thinking partner для Стадии 2.** Claude не просто фиксирует ответы собственника, а задаёт «а почему?», challenge-ит противоречия. *Источник: Елкин.*
- **Фидбэк клиента не 1-в-1 в промпт.** Правки от собственника идут через шаблон-обёртку: поля «что отверг», «почему», «как переформулировать». Сырой фидбэк не уходит в Anthropic. *Источник: Горшков.*
- **Cross-check между командами.** Выход `/values-draft` становится input для `/positioning-draft`; `/positioning-draft` — input для `/message-variants`. Обеспечивается через структуру `Row` → `Draft` → `Finalized` в PG. *Источник: синтез.*
- **Borderline classify green/yellow/red** (`/review-classify`). Любой output на границе валидатора (месседж ровно 4/7 слов; миссия с «маркером») прогоняется через отдельный classifier. Красные блокируются, жёлтые требуют явного клика маркетолога. *Источник: курс, файл 11.*

### Анти-паттерны prompt engineering

- «Чем сложнее промпт, тем лучше» — неверно. Промпты команд максимально конкретны и минимально словесны. *Источник: курс, файл analysis (5).*
- Один длинный чат на всё. Для BP — одна задача = один запрос.
- Изменение промпта без прогона golden set (блокирует CI).

## 4. Output validator и golden set

### MVP

- **3-уровневая система доверия:** (1) regex — жёсткие правила (4-7 слов в бренд-месседже, запрет пустых полей); (2) LLM-judge — семантика (тон ценности соответствует архетипу, позиционирование содержит 5 переменных формулы); (3) human review маркетолога; (4) approval собственника кнопкой. *Источник: курс, файл 10.*
- **Methodology-compliance-check.** Отдельный LLM-judge с прямой ссылкой на канон 3.1 (`Brend_platforma_3_1.docx`). Ловит логические расхождения — например, «миссия сгенерена для B2B вместо offline B2C», «архетип не из 6 разрешённых». Обычный quality-judge этого не ловит. *Источник: Щербатюк (дельта) + синтез.*
- **Golden set — 3-5 эталонов.** Стартуем с Белая Линия; добавляем 2-4 эталона из пилотов (обязательно разные индустрии — мебель / ресторан / салон). *Источник: Твердохлебов.*
- **Walk-forward regression, не monolith.** Golden set прогоняется на «слепых» изолированных срезах: отдельно ценности, отдельно миссия, отдельно архетип. Избегает переобучения промптов под один эталон. *Источник: курс, файл 01.*
- **CI-триггер:** любое изменение `prompts/` или `industry_context/`. Деградация LLM-judge >15% по embedding similarity — блокирует релиз промпт-шаблона. *Источник: Твердохлебов + курс.*

### Пример ошибки, которую ловит methodology-compliance-check

> Claude сгенерировал ценность «инновационный подход» для стоматологии «Белая Линия». Regex пропустил (нет стоп-слов). Quality-judge пропустил (связный текст). Methodology-compliance-check заблокировал: «абстракция, нарушает требование конкретности эмоциональных слов (канон 3.1, раздел "Ценности")».

## 5. Безопасность и reseller-модель (дельта к [PLAN.md §8](./PLAN.md))

### 🔴 Критически упущенное в текущем PLAN.md

- **`BriefSanitizerService` input-sanitizer до первого Anthropic-вызова.** В плане есть только output-validator. Prompt injection — **TOP-1 уязвимость LLM-приложений**. Компоненты: LLM-judge + regex-чёрный список (`ignore previous`, `system:`, markdown-инъекции, `<|...|>` делимитеры). Логирование срабатываний в `security_events`. *Источник: Щербатюк — «уязвимость №1».*
- **Escaping делимитерами:** весь пользовательский ввод маркетолога оборачивается в `<user_input>...</user_input>` теги, Claude инструктируется не следовать директивам внутри них. Jailbreak-detection в validator. Никогда не echo'ить системный промпт в output. *Источник: курс, файл 08 — Lethal Trifecta.*
- **PII auto-redaction.** Пре-скрининг брифа на паспорта сотрудников, номера карт, ФИО частных лиц, адреса. Regex `[A-Za-z0-9]{32,}` для похожего на токены. Warning маркетологу при загрузке. *Источник: Щербатюк.*
- **Sandbox tool-calls LLM.** Whitelist tool-names в ответе Anthropic (например, только `draft_value`, `generate_mission`, `classify_message`), отклонять всё остальное. *Источник: Щербатюк — «агенты сносили папочки».*
- **Immutable snapshot на `owner_approved`.** Автоматический snapshot артефакта (ценности / миссия / месседж) в cold storage (S3 Glacier) при нажатии кнопки собственником. Защита от «утверждение было, но документ изменился». *Источник: Щербатюк — «правило №0 — бэкапы».*
- **TLS 1.3 + mTLS** между NestJS ↔ Anthropic, PG в transit тоже шифруется. В текущем плане только rest (AES-256-GCM). *Источник: Щербатюк.*
- **`docx-exporter` — отдельный Docker-контейнер** с read-only root, запись только в `/tmp/export-{uuid}`. Никакого `exec`, никакого доступа к сети, кроме S3-upload. *Источник: Щербатюк — whitelist pattern.*
- **GitHub push hygiene.** Pre-commit hook с `gitleaks` для `prompts/*.md` (промпт-шаблоны часто содержат примеры с чувствительными данными). Запрет force-push в `main` и в `main/prompts`. *Источник: Щербатюк.*
- **SAST + DAST в CI.** Semgrep (SAST) на backend + OWASP ZAP (DAST) на staging перед релизом. *Источник: Щербатюк.*
- **Shadow AI политика для `chip_manager`.** Корпоративный Claude-workspace + правило «черновики BP не покидают BP» (запрет копировать тексты в личные ChatGPT/Gemini). Документируется в договоре с трекером. *Источник: Щербатюк.*
- **No Notion/Slack MCP в MVP.** Только Anthropic API, никаких интеграций с Notion/Slack/Google Docs. Зафиксировать в CLAUDE.md явно. *Источник: Твердохлебов.*
- **Human в цепочке ответственности.** В `audit_events` обязательные поля: `responsible_user_id`, `generated_by` (`claude-opus-4.7`), `modified_by` (`user_id` маркетолога), `approved_by` (`user_id` собственника). Без полного комплекта подписей — DOCX-экспорт блокируется. *Источник: Щербатюк + курс, файл 07.*

### Что уже есть в PLAN.md §8 и подтверждается

- Ключи только у Чиркова, ротация 1 минуту — ✅ подтверждается.
- Server-to-server only, промпты не в UI — ✅ подтверждается.
- Retention 14 дней debug-логов — ✅, дополняется PII-pre-screen.
- AES-256-GCM на rest — ✅, дополняется TLS 1.3 в transit.

### Анти-паттерны безопасности

- `ANTHROPIC_API_KEY` в репо / `.env` в git (обязательно в `.gitignore`: `.env`, `*.env.local`, `credentials.json`). *Источник: курс, файл 07.*
- Audit log с сырым API-ключом — никогда.
- Полный доступ tool-calls («утилита может всё, что хочет Claude») — always deny by default.

## 6. UX маркетолога (этапы wizard'а)

### MVP

- **Чек-листы до «Next».** Wizard проверяет, что маркетолог заполнил все обязательные поля стадии; не даёт пропускать шаги. *Источник: Замесин — «нейронки в пайплайне не ленятся».*
- **UX-суфлёр.** Справа в wizard-UI панель «подсказки» от LLM-judge по текущему полю. Пример: «в этой ценности нет конкретного проявления, Claude предлагает: "перезваниваем после визита в течение 24 часов"». *Источник: Доронин — «суфлёр для продажника».*
- **Чип «Claude сделал черновик за 45 сек — вы сэкономили N часов».** Ключ к закреплению привычки принимать/править вместо переписывания с нуля. Счёт времени на основе бенчмарка ручного заполнения (~2-3 часа на `/values-draft`). *Источник: Хегай — «страх делегировать — главный барьер».*
- **Onboarding-видео «вы не кодите, вы редактируете».** 30 секунд первого запуска, фиксирует роль маркетолога как методолога. *Источник: Хегай — «я гуманитарий».*
- **Никаких JSON / markdown / API-ключей в UI.** Только визарды, дропдауны, текстовые поля. Любая техническая настройка — не доходит до маркетолога. *Источник: Хегай.*
- **Форма правок как шаблон, не свободный текст.** Маркетолог отвергает черновик → форма с полями «что отверг» / «почему» / «как переформулировать». Эти три поля уходят в промпт переделки. *Источник: Горшков.*
- **Fail-safe при падении Anthropic.** Баннер + кнопка «заполнить вручную», работа сохраняется.

### Неожиданный инсайт про маркетолога ЧиП

> Маркетолог переписывает 80% AI-черновиков с нуля, потому что не доверяет. Правильный UX — не «улучши promt», а «заставь принять черновик как стартовую точку». Отсюда чип «сэкономил N часов» и onboarding-видео. *Источник: Хегай inner insight.*

## 7. Usage UI и тарифы (дельта к [PLAN.md §7](./PLAN.md))

### MVP

- **Тарифы по уровню ручного ревью Чиркова:**
  - **Economy:** 0 часов ручного ревью, автовалидация, SLA 48ч.
  - **Standard:** 2 часа на проект (ключевые точки: ценности, миссия, месседж), SLA 24ч.
  - **Premium:** unlimited + offline-встреча Чиркова с маркетологом и собственником, SLA 4ч.
  *Источник: Петров — «консультант, который перестал ждать разработчиков».*
